# Enter The Dungeon PWA Prototype

Progressive Web App prototype focused on offline support and a direct CSR vs SSR comparison.

## Overview

This project is a React + Vite PWA called **Enter The Dungeon**. It includes:

- Character archive UI with search, detail view, and edit flow.
- Launch splash screen (implemented in React + CSS).
- Service worker registration and offline-capable caching strategies.
- Two rendering modes for comparison:
  - **CSR** route: `/`
  - **SSR** route: `/ssr`

## Features Implemented

### PWA Features

- Web app manifest configured with name, theme colors, and install settings.
- Service worker generated by `vite-plugin-pwa`.
- Offline behavior through Workbox runtime caching:
  - SPA navigation and static assets.
  - API data (`/api/characters`).
  - SSR page caching (`/ssr`) after first online visit.

### Splash Screen

- A launch splash screen appears on app startup before the main UI.
- Designed with custom CSS and loading animation.

### Rendering Modes

- **CSR (`/`)**: React app renders in the browser, fetches data client-side, and supports interactive navigation without full reloads.
- **SSR (`/ssr`)**: Server returns pre-rendered HTML with character data for faster initial content delivery.

## How to Run Locally

### Prerequisites

- Node.js 18+ (recommended latest LTS)
- npm

```bash

```

- `http://localhost:5173/` for CSR
- `http://localhost:5173/ssr` for SSR

```bash
npm run build

```

## CSR vs SSR Findings

### Conceptual Difference

- **CSR** sends a mostly empty HTML shell and renders content in the browser via JavaScript.
- **SSR** sends already-rendered HTML from the server, so meaningful content appears earlier.

### Practical Impact in This Project

- CSR gives richer client interaction and strong offline behavior through cached assets/data.
- SSR improves initial delivery cost and network footprint for first paint/useful content.

## Performance Results

Measured comparison:

### CSR

- 42 requests
- 12.4 kB transferred
- 2.2 MB resources
- Finish: 1.35 s
- DOMContentLoaded: 122 ms
- Load: 143 ms

### SSR

- 14 requests
- 15.5 kB transferred
- 599 kB resources
- Finish: 400 ms
- DOMContentLoaded: 327 ms
- Load: 358 ms

### Interpretation

- SSR completed significantly faster overall and used fewer requests/resources.
- CSR had faster DOMContentLoaded/Load events in this run, but higher total resource cost.
- Both modes are useful depending on goals: SSR for initial delivery efficiency, CSR for rich client interaction and offline-first app behavior.

## Challenges and Solutions

1. **Combining CSR + SSR in one project**
	- Challenge: Keep React SPA behavior while adding true SSR output.
	- Solution: Added dedicated SSR endpoint (`/api/ssr`) and route (`/ssr`) while preserving SPA routing at `/`.

2. **Offline behavior differences between CSR and SSR**
	- Challenge: SSR cannot fetch fresh server content when fully offline.
	- Solution: Added runtime cache for `/ssr` so previously visited SSR pages can load offline as cached snapshots.

3. **Environment execution policy (Windows PowerShell)**
	- Challenge: `npm.ps1` blocked by script execution policy in some environments.
	- Solution: Used `npm.cmd` for command execution when needed.

## Notes

- For SSR offline fallback to work, `/ssr` must be visited at least once while online.
- The primary PWA offline experience is still strongest on the CSR route (`/`) due to asset/data caching.
